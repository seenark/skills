# Complete Guide Course Builder Refinement Design

## เป้าหมาย

ปรับ `complete-guide-course-builder` ให้สร้างคอร์สภาษาไทยที่ผู้เรียนเปิดอ่านและลงมือทำตามได้ตั้งแต่ต้นจนจบ โดยไม่ต้องเปิด source repository คู่กันระหว่างเรียน

## ปัญหาปัจจุบัน

1. กฎภาษาใช้คำว่า “primarily in Thai” จึงยังเปิดช่องให้หัวข้อและคำอธิบายทั่วไป เช่น `Common Mistakes`, `Solution`, `solution walkthrough` เป็นภาษาอังกฤษ
2. บทเรียนอ้าง path และแนวคิดจาก repository แต่ไม่ได้รับประกันว่าจะฝัง code และ file state ที่จำเป็นไว้ในบทเรียน
3. กฎ “small code block” ช่วยเรื่องความเข้าใจ แต่ไม่กำหนด final state ของไฟล์ ทำให้ผู้เรียนอาจประกอบ code ไม่ครบ
4. เนื้อหาอาจดูเหมือนข้อความแปล หากบังคับแปลศัพท์ทุกคำแทนการเขียนภาษาไทยธรรมชาติแบบผู้สอน

## ข้อกำหนดภาษา

- ร้อยแก้ว คำอธิบาย หัวข้อบทเรียน แบบฝึกหัด เฉลย ข้อผิดพลาดที่พบบ่อย และบทสรุป ต้องเขียนเป็นภาษาไทยธรรมชาติ
- คง technical terms ภาษาอังกฤษเมื่อเป็นคำที่นักพัฒนาไทยใช้จริง เช่น component, route, service, repository, API, request, response, validation, refactor, test, debug, build, deployment และชื่อ symbol/API จาก code
- ไม่แปล technical terms แบบตรงตัวจนเกิดคำไทยที่ไม่เป็นธรรมชาติ
- ภาษาไทยต้องเกิดจากการอธิบายแนวคิด ไม่ใช่แปลประโยคภาษาอังกฤษทีละคำ
- ใช้ตัวอย่าง การเปรียบเทียบ mental model และบริบทการทำงานเมื่อช่วยให้เข้าใจ แต่ไม่ยัด analogy ที่ไม่จำเป็น
- ชื่อไฟล์ path คำสั่ง CLI code identifier และข้อความที่โปรแกรมต้องใช้จริงคงรูปเดิม

## รูปแบบบทเรียนแบบ Self-Contained

ทุกบทต้องมีข้อมูลเพียงพอให้ผู้เรียนทำตามโดยไม่เปิด source repository คู่กัน:

1. ระบุ branch/checkpoint เริ่มต้น
2. แสดงโครงสร้างไฟล์ที่เกี่ยวข้องก่อนเริ่ม
3. อธิบายปัญหาและ mental model เป็นภาษาไทย
4. ระบุ path ก่อน code block ทุกครั้ง
5. แสดงคำสั่งติดตั้ง สร้างไฟล์ run build test หรือ debug ที่ต้องใช้ตามลำดับ
6. แสดง code ทีละส่วน พร้อมอธิบายหน้าที่ เหตุผล ผลเมื่อเอาออก และข้อผิดพลาดที่พบบ่อย
7. เมื่อใช้ code จาก upstream repository ให้คัดเฉพาะส่วนที่จำเป็นมาไว้ในบท พร้อมระบุ source path และ commit/tag ที่วิเคราะห์
8. ห้ามสั่งให้ผู้เรียน “เปิดไฟล์ใน repository แล้วดู...” โดยไม่แสดงเนื้อหาที่ต้องดูในบท
9. ท้ายบทแสดง `Final File State` ซึ่งรวมเนื้อหาเต็มล่าสุดของทุกไฟล์ที่สร้างหรือแก้ในบท
10. ระบุคำสั่งตรวจสอบและผลลัพธ์ที่คาดหวัง
11. ให้แบบฝึกหัดและเฉลยครบ
12. ระบุ finished checkpoint

## การจัดการ Code ขนาดใหญ่

- ระหว่างสอนแบ่ง code เป็นช่วงเล็กเพื่ออธิบายได้ชัด
- ท้ายบทต้องรวม full file state ของไฟล์ที่แตะ แม้จะมี code ซ้ำจากช่วงอธิบาย
- หากไฟล์ต้นฉบับใหญ่มาก ให้สร้าง teaching version ที่เล็กแต่ runnable ก่อน แล้วค่อยแสดงเฉพาะ upstream excerpt ที่จำเป็นพร้อมบริบทครบ
- ห้ามตัด code ที่จำเป็นต่อการ compile, run หรือ test แล้วปล่อยให้ผู้เรียนไปหาเอง
- ไฟล์ที่ไม่ได้แตะในบทไม่ต้องทำซ้ำ แต่ต้องบอก dependency ที่บทนี้ใช้และชี้ไปยังบทก่อนหน้าที่มี full state

## โครงสร้างหัวข้อภาษาไทย

หัวข้อบังคับของบทเรียนจะใช้ภาษาไทยที่อ่านเป็นธรรมชาติ โดยคง technical terms ที่จำเป็น เช่น:

- เป้าหมายของบทเรียน
- เริ่มจากปัญหา
- Mental model
- ตัวอย่างที่เล็กที่สุด
- วิธีตรงไปตรงมาแบบแรก
- ข้อจำกัดของวิธีแรก
- ปรับเป็นวิธีที่ดูแลต่อได้ง่ายขึ้น
- เชื่อมกับ code ใน repository จริง
- ข้อผิดพลาดที่พบบ่อย
- บริบทการทำงานจริง
- ใช้ AI ช่วยเขียน code อย่างเข้าใจ
- แบบฝึกหัด
- เฉลยแบบละเอียด
- สรุปท้ายบท
- สภาพไฟล์ฉบับสมบูรณ์ท้ายบท
- วิธีตรวจสอบผล

ชื่อ directory เช่น `solutions/` และ `common-mistakes/` คงไว้เพื่อความสม่ำเสมอของ output contract แต่ชื่อหัวข้อและเนื้อหาภายในต้องเป็นภาษาไทยธรรมชาติ

## Supporting Files

- `exercises/`: โจทย์ภาษาไทย พร้อมเป้าหมาย จุดเริ่มต้น งานที่ต้องทำ expected behavior คำใบ้ และบทที่เกี่ยวข้อง
- `solutions/`: เฉลยภาษาไทยแบบละเอียด พร้อม code เต็ม เหตุผล วิธีตรวจสอบ และตัวอย่างวิธีที่ผิด
- `common-mistakes/`: อธิบายอาการ สาเหตุ วิธี debug วิธีแก้ และวิธีป้องกันเป็นภาษาไทย
- `final-project/`: โจทย์ acceptance criteria branch plan คำใบ้ และเฉลย/solution walkthrough เป็นภาษาไทย โดยใช้ technical terms ตามธรรมชาติ
- `ai-prompts/`: prompt อาจเป็นภาษาไทยหรืออังกฤษตามวัตถุประสงค์ แต่คำอธิบาย วิธีใช้ และคำเตือนต้องเป็นภาษาไทย

## การทดสอบ Skill

### RED

ทดสอบ skill ปัจจุบันด้วย scenario ที่ขอคอร์สจาก repository ขนาดเล็กและตรวจว่า agent:

- ใช้หัวข้ออังกฤษในส่วนที่ไม่ใช่ technical term หรือไม่
- อ้างให้เปิด repository เองหรือไม่
- แสดง final state ของทุกไฟล์ที่แตะหรือไม่
- ให้คำสั่ง run/test และ expected output หรือไม่
- เขียนภาษาไทยแบบแปลตรงตัวหรือไม่

### GREEN

หลังแก้ skill ให้รัน scenario เดิมและตรวจว่า output contract ระบุครบ:

- prose และหัวข้อทั่วไปเป็นไทย
- technical terms คงเป็นอังกฤษอย่างเป็นธรรมชาติ
- ทุกบท self-contained
- มี file tree, path, commands, incremental code และ full final state
- มี verification, exercise และเฉลยครบ

### REFACTOR

ตรวจ loophole เช่น “อ้างว่า code ยาวเกินไป”, “ให้ลิงก์แทน code”, “มีเฉลยแต่ไม่แสดง full code”, หรือ “แปลหัวข้อเป็นไทยแต่เนื้อหายังเป็นอังกฤษ” แล้วเพิ่ม counter เฉพาะที่พบจริง

## เกณฑ์ยอมรับ

- ผู้เรียนทำบทเรียนครบได้โดยไม่เปิด source repository คู่กัน
- ผู้เรียนสร้าง code และ file state ที่บทกำหนดได้จากเนื้อหาในคอร์สเพียงอย่างเดียว
- ภาษาไทยอ่านเหมือนผู้สอนชาวไทยอธิบาย ไม่เหมือน machine translation
- technical terms ที่ใช้จริงไม่ถูกแปลจนแปลก
- ทุกไฟล์ที่แตะในบทมี full final state
- ทุกบทมีคำสั่งตรวจสอบและ expected result
- `solutions/`, `common-mistakes/` และ final-project solution walkthrough ใช้คำอธิบายภาษาไทยครบ
